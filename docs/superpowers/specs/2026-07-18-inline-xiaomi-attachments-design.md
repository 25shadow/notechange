# 小米附件按正文顺序预览设计

## 目标

本地导出批次预览应按照小米笔记正文中的附件占位符顺序展示图片，而不是把所有图片统一堆放在正文末尾。同时，旧版小米富文本中的 `<b>` 和 `<size>` 不再被错误标记为无法自动迁移。

## 内容规则

1. 小米正文使用 `☺ <fileId>`（后面可带 `<.../>` 元数据）表示附件位置。
2. 预览详情必须暴露附件的 `sourceId`，用它与正文中的 `fileId` 精确匹配。
3. 匹配到 `image/*` 附件时，在占位符所在位置渲染图片，并隐藏原始占位符文本。
4. 匹配到非图片附件时，在原位置显示一个文件条目，不把音频、手写数据或其他文件冒充图片。
5. 没有被正文引用的附件保留在正文末尾的“其他附件”区域，避免数据静默丢失。
6. 同一附件只渲染一次；找不到附件的占位符保留为普通文本，便于人工核对。

## 富文本兼容

- `<b>` 转换为标准 `<strong>`，保留粗体语义。
- `<size>` 去掉标签外壳但保留全部子内容；小米旧标签没有稳定、可验证的字号映射，因此不猜测具体字号。
- 上述两种标签不再生成 `unsupported-content` 警告。
- 其他尚未支持的标签仍按现有规则清洗并生成警告。

## 数据流与组件边界

主进程在 `ExportPreviewDetail.attachments` 中增加 `sourceId`，只传递匹配所需的稳定标识，不传本地绝对路径。渲染进程将正文拆分为“文本片段”和“附件片段”，附件仍通过现有 `getExportAttachment` IPC 按需读取为 data URL。正文拆分逻辑做成独立纯函数，预览组件只负责渲染。

## 错误处理

- 图片读取失败时在原位置显示“附件读取失败”，其余正文继续显示。
- 非图片附件显示文件名和类型，不尝试使用 `<img>` 解码。
- 重复或失配的占位符不导致预览崩溃。

## 测试

1. 内容规范化测试覆盖 `<b>` 转 `<strong>`、`<size>` 保留文字且不报警。
2. 正文拆分测试覆盖多张图片的原始顺序、未引用附件、未知占位符和非图片附件。
3. 预览组件测试确认图片出现在相邻正文段落之间，而不是只出现在末尾。
4. 完整测试、类型检查、契约校验与生产构建继续作为完成门槛。

## 非目标

- 不改变本地导出文件格式和既有批次数据。
- 不猜测小米 `<size>` 的视觉字号。
- 本次不实现 vivo 附件上传。
